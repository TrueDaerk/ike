# Roadmap 0100 — LSP Support

This roadmap gives IKE real language intelligence by speaking the
[Language Server Protocol](https://microsoft.github.io/language-server-protocol/).
It implements a JSON-RPC 2.0 client that spawns and manages one or more language
servers over stdio, drives text-document synchronisation from the vim editor's
edit events (06), and renders diagnostics, completion, hover, go-to-definition and
the rest back into the editor pane. It also defines a fast lexical syntax
highlighter (Tree-sitter) layered with optional LSP semantic-token highlighting.

Everything async respects the bubbletea event loop (01): no LSP I/O ever blocks
`Update`. Server traffic runs in goroutines and results re-enter the program as
`tea.Msg`s produced by `tea.Cmd`s. LSP-driven actions (go-to-definition, rename,
format, code action, …) are registered as `Command` capabilities through the
plugin registry (02), so the command palette (07) and the keybinding layer (08)
reach them through the single, shared mechanism. All server configuration is read
from the `[lsp]` section of `internal/config` (04), whose *content* this roadmap
specifies; theme colours for diagnostics and highlighting come from `[theme]` (04).

## Prerequisites / Dependencies

- **01 Foundation** — the bubbletea program and the `tea.Msg`/`tea.Cmd`
  async pattern. LSP I/O is wrapped in `tea.Cmd`s; responses, notifications and
  crash signals flow back as `tea.Msg`s routed to the editor pane. Nothing here
  blocks `Update`.
- **02 Plugins registry** — `internal/plugin` (`Command`, `Keymap`, `Pane`,
  `FileHandler`, `Hook`), `internal/registry`, `internal/host` (`host.API`). LSP
  actions register as `Command` capabilities; the LSP subsystem is wired in
  through `host.API` like any other capability provider.
- **04 Settings** — `internal/config` owns the loader/format/reload. This roadmap
  defines the *content* of the `[lsp]` section (server table, enable flags,
  per-server settings) and consumes it; it does not own the loader. Diagnostic and
  highlight colours are pulled from `[theme]`.
- **06 Vim editor** — owns the `Buffer`, modal engine, and the integration seams
  in `internal/editor/events.go` (on-change, cursor-move, completion-trigger) plus
  viewport/cursor info. LSP **consumes** those seams: edit events drive text sync;
  diagnostics/completion/hover render into the editor. The buffer is never
  reimplemented here.

Soft / forward dependencies (seams only): **07** palette UI, **08** keybinding
defaults — both reach LSP actions purely via registered `Command` IDs.

## Architecture

```
internal/lsp/
  lsp.go               # subsystem entry; wires manager + features into host.API
  config.go            # [lsp] schema CONTENT: ServerSpec table, enable flags,
                       #   per-server settings; resolves language/extension -> spec
  jsonrpc/
    conn.go            # JSON-RPC 2.0 conn over an io.ReadWriteCloser (stdio)
    framing.go         # Content-Length header framing (read/write LSP messages)
    message.go         # Request/Response/Notification, id allocation, error codes
    conn_test.go
  transport/
    stdio.go           # spawn process (cmd+args+env+cwd), pipe stdin/stdout/stderr
    process.go         # lifecycle: start, stop, exit watch, stderr capture
  client/
    client.go          # one Client per running server: typed request/notify API
    lifecycle.go       # initialize/initialized/shutdown/exit + capability cache
    capabilities.go    # negotiated ServerCapabilities; feature gating helpers
    restart.go         # crash detection + backoff restart, re-open tracked docs
  manager/
    manager.go         # Manager: language/root -> Client, spawn on demand, route
    roots.go           # workspace-root detection from config root markers
    registry_bridge.go # registers LSP actions as registry Commands (the only bridge)
  protocol/
    types.go           # LSP structs (Position, Range, Location, Diagnostic, ...)
    enums.go           # CompletionItemKind, SymbolKind, DiagnosticSeverity, ...
    convert.go         # editor Position/Range  <->  LSP (UTF-16) position mapping
    convert_test.go
  sync/
    sync.go            # consumes editor edit events -> didOpen/didChange/didClose
    incremental.go     # incremental TextDocumentContentChangeEvent builder
    version.go         # per-document version counter; didSave seam
    sync_test.go
  feature/
    diagnostics.go     # store publishDiagnostics; expose to editor + status line
    completion.go      # textDocument/completion -> editor completion-seam items
    hover.go           # hover request -> popup content
    definition.go      # go-to-definition (+declaration/type-definition)
    references.go      # find references -> results list
    rename.go          # prepareRename + rename -> WorkspaceEdit apply
    signature.go       # signatureHelp -> popup
    formatting.go      # formatting / rangeFormatting -> buffer edits
    codeaction.go      # codeAction -> quick-fixes / refactors
    workspace_edit.go  # apply a WorkspaceEdit across buffers via editor seams
    *_test.go
  messages.go          # tea.Msg types: Diagnostics, Completion, Hover, Definition,
                       #   References, ServerCrashed, ... ; tea.Cmd constructors

internal/highlight/
  highlight.go         # highlight subsystem; merges syntax + semantic layers
  syntax/
    treesitter.go      # Tree-sitter parser per language; incremental reparse
    grammars.go        # grammar registry (language -> grammar), lazy load
    query.go           # highlight queries -> spans; theme-keyed capture names
    syntax_test.go
  semantic/
    tokens.go          # LSP semantic tokens (full + delta) -> spans (optional)
    legend.go          # decode SemanticTokensLegend into capture-like names
  span.go              # HighlightSpan{Range, Capture}; theme lookup from [theme]
  merge.go             # layer precedence: syntax base, semantic overlay, diags
```

`manager.Manager` is the single owner of running servers: it maps
(language, workspace root) to a `client.Client`, spawning servers lazily on the
first relevant `didOpen`. `sync` translates editor edit events into document
notifications; `feature/*` turn editor requests into `tea.Cmd`s and server
replies into `tea.Msg`s. `registry_bridge.go` is the only place LSP touches
`internal/registry`. Highlighting is a sibling subsystem: Tree-sitter provides a
fast lexical base layer immediately, and LSP semantic tokens overlay it when the
server supports them.

## Design rules

- **Never block the event loop.** Every server interaction is launched as a
  `tea.Cmd` and resolves through a `tea.Msg`. The `jsonrpc` conn runs read/write
  loops on goroutines; the model only ever sees messages. `Update` does no I/O.
- **One manager owns all servers.** Spawning, routing, capability gating and
  restart live in `manager`/`client`. No feature spawns a process or talks to a
  raw connection directly; features go through the typed `client.Client` API.
- **Position mapping is centralised.** LSP positions are UTF-16 code-unit based;
  the editor is rune/byte based (06). `protocol/convert.go` is the *only* place
  that converts, honouring the server's negotiated `positionEncoding` when offered.
- **Text sync is event-driven, not polled.** `sync` reacts to the editor's
  on-change seam (06): `didOpen` on file open, incremental `didChange` per edit
  (full-text fallback if the server lacks incremental sync), `didSave`, `didClose`
  on close. Document versions are monotonic and owned by `sync/version.go`.
- **Capabilities gate features.** A feature is only offered (and only registered as
  a usable `Command`) when the server advertises it. Missing capability = graceful
  no-op, never an error popup.
- **Crashes are recoverable.** `client/restart.go` detects process exit, restarts
  with backoff, re-runs `initialize`, and re-opens every tracked document so the
  user keeps working. Repeated crashes disable the server with a status message.
- **Actions are registry Commands, with stable IDs.** Go-to-definition, find
  references, rename, format, code action, restart-server, etc. are plain
  functions wrapped by `registry_bridge.go` as `Command` capabilities. The palette
  (07) and keybindings (08) invoke by ID — LSP grows no parallel dispatch path.
- **Config is content here, mechanism is 04.** This roadmap defines the `[lsp]`
  shape — a server table keyed by language/extension with `command`, `args`,
  `env`, `root_markers`, an `enable` flag, and a free-form per-server `settings`
  blob forwarded as `initializationOptions`/`workspace/didChangeConfiguration`.
  The loader and reload come from `internal/config`.
- **Highlighting: Tree-sitter first, semantic tokens to augment.** Lexical
  highlighting via Tree-sitter is fast, synchronous-friendly (incremental
  reparse), and works without a server — it is the base layer and ships first. LSP
  semantic tokens are an *optional overlay* added later for servers that support
  them; `merge.go` defines precedence (syntax base < semantic overlay < diagnostic
  underlines). All colours are resolved from `[theme]` (04); this roadmap defines
  no palette.
- **Tests ship with code.** `jsonrpc` framing, position conversion, incremental
  change building, capability gating, and highlight span/merge logic are unit
  tested with fakes (a scripted in-memory server) so no real binary is required.

## Milestones

- [x] JSON-RPC 2.0 core in `internal/lsp/jsonrpc`: Content-Length framing,
      request/response/notification, id allocation, error codes, async read/write
      loops over an `io.ReadWriteCloser`.
- [x] Transport in `internal/lsp/transport`: spawn a server process over stdio
      (command, args, env, cwd) with stderr capture and exit watching.
- [x] Client lifecycle in `internal/lsp/client`: `initialize` / `initialized` /
      `shutdown` / `exit`, capability negotiation, and a cached
      `ServerCapabilities` with feature-gating helpers.
- [x] Manager in `internal/lsp/manager`: map (language, workspace root) -> client,
      detect roots from config `root_markers`, spawn lazily, route messages;
      support multiple concurrent servers by language.
- [x] `[lsp]` config CONTENT in `internal/lsp/config.go`: server table
      (language/extension -> command + args + env + root markers), enable flags,
      and per-server `settings` forwarded as init options (defaults for
      gopls/intelephense/pyright registered via `config/extend.go`; loader owned by 04).
- [x] Position mapping in `internal/lsp/protocol/convert.go`: editor rune/byte
      positions <-> LSP UTF-16 positions, honouring negotiated `positionEncoding`.
- [x] Text-document sync (in `internal/lsp/manager`): consume editor edit events
      (06) to send `didOpen`, `didChange` (full-text for the MVP; incremental
      deferred), `didSave`, `didClose`, with monotonic per-document versions.
- [x] Diagnostics: store `publishDiagnostics`, deliver as `tea.Msg`, and render in
      the editor gutter (severity colour) + inline underline + a status-line summary.
- [x] Completion: `textDocument/completion` driven by the editor completion-trigger
      seam (06), rendered in a cursor-anchored popup with prefix filtering + accept.
- [x] Go-to-definition: definition (+ LocationLink), navigating the editor to the
      target `Location` and placing the cursor.
- [x] Hover: `textDocument/hover` rendered in a popup at the cursor.
- [x] Crash recovery in `internal/lsp/manager/restart.go`: detect exit, backoff
      restart, re-`initialize`, re-open tracked documents; disable after repeated
      failures with a status message.
- [x] Tree-sitter syntax highlighting in `internal/highlight`: per-language grammar
      registry (Go/PHP/Python), parse-on-change as a `tea.Cmd`, highlight-query
      spans keyed to `[theme]` captures — the fast lexical base layer (CGo, behind a
      build tag with a no-op stub for `CGO_ENABLED=0`).
- [ ] Signature help: `textDocument/signatureHelp` popup while typing arguments. *(deferred)*
- [ ] Find references: `textDocument/references` into a results list/pane. *(deferred)*
- [ ] Rename: `prepareRename` + `rename` applying the returned `WorkspaceEdit`
      across buffers via the editor seams (`feature/workspace_edit.go`). *(deferred)*
- [ ] Formatting: document and range formatting applied as buffer edits. *(deferred)*
- [ ] Code actions: `textDocument/codeAction` quick-fixes / refactors, applying
      their `WorkspaceEdit`s. *(deferred)*
- [ ] LSP semantic-token highlighting overlay in `internal/highlight/semantic`
      (full + delta), merged over the Tree-sitter base per `merge.go` precedence
      (optional, capability-gated). *(deferred)*
- [x] LSP actions exposed as registry `Command`s (stable IDs) via the `plugins/lsp`
      bridge — `lsp.hover`, `lsp.definition`, `lsp.restart` — reachable by palette
      (07) and keybindings (08); no parallel dispatch path. (references/rename/format/
      codeaction commands land with their deferred features.)
- [x] Tests: table-driven and fake-server coverage for JSON-RPC framing, position
      conversion, capability gating, manager routing/diagnostics/completion, restart
      logic, highlight span math, and the editor diagnostics/completion/hover state.
- [x] Wiki: add/refresh the LSP and highlighting concept doc(s) under `wiki/`
      (frontmatter with `type`, `title`, `description`, `resource` ->
      `internal/lsp` and `internal/highlight`), refresh `timestamp`, and add a
      `log.md` entry.

## Out of scope

- The editor buffer and modal engine — **Roadmap 0060** (consumed via its seams).
- The config loader, file format and reload mechanism — **Roadmap 0040** (this
  roadmap only defines the `[lsp]` content and consumes it).
- The command palette UI — **Roadmap 0070** (reaches LSP via `Command` IDs).
- Default keybinding tables / user keymap config — **Roadmap 0080**.
- Theme colour definitions — `[theme]` in **Roadmap 0040** (highlighting only
  *resolves* captures to those colours; it defines no palette).
- Debug Adapter Protocol (DAP) / debugging, test runners, and build-tool
  integration — possible later roadmaps; not LSP.
- Multi-root / workspace-folder federation beyond per-file root detection, and
  call hierarchy / inlay hints / document symbols outliner — later increments;
  the manager and capability model are shaped to allow them.
